CURVE DATA DECOMPRESSED FORMAT
================================================================================
Stored in Curve_data ($00FF5B00), read by Update_slope_data / Integrate_curveslope.
One byte per step forward. Ends with $FF ($80+).
Code reads 40 bytes ahead; ca 26 steps are rendered.

Byte values:
* $00       = Straight
* $01-$2F   = Left turn, from extreme (01) to soft (2F)
* $30-$3F   = Game freezes (invalid)
* $41-$6F   = Right turn, from extreme (41) to soft (6F)
* $70-$7F   = Game freezes (invalid)
* $80-$FF   = Sentinel / wrap: continue reading from track start (new lap)

Lower 6 bits = sharpness (higher = softer). Bit 6 = right turn. Bit 7 = wrap sentinel.

Background horizontal displacement ($00FF6300 = Background_horizontal_displacement):
Encoded together with curve data in ROM as relative delta values.
In memory stored as absolute rotation. Update_slope_data integrates via ADD/SUB each frame.


SLOPE DATA DECOMPRESSED FORMAT (SOLVED — see Update_slope_data header in smgp.asm)
================================================================================
Visual slope:  Visual_slope_data  ($00FF7300) — drives background vertical shift
Physical slope: Physical_slope_data ($00FF8300) — drives RPM drag via Track_phys_slope_value

Byte encoding identical to curve data:
* $00       = Flat
* $01-$2F   = Down slope (extreme → soft)
* $41-$6F   = Up slope (extreme → soft)
* $80+      = Sentinel (BMI branch terminates RLE loop)

ROM encoding: 3/4-byte RLE records (initial displacement byte + per-record count + delta).
Physical slope uses simpler 3-byte RLE. Both terminated by high-byte bit 7 set.
Track_phys_slope_value is used by Update_rpm: negative = uphill drag on engine.


TRACK DATA VALIDITY CONSTRAINTS (DOC-008)
================================================================================
These rules govern what constitutes a valid track in Super Monaco GP.
Violating them causes crashes, visual glitches, or wrong gameplay.
Source: driving.asm, rendering.asm, race_support.asm, ai.asm, road_and_track_data.asm,
        track_config_data.asm.

--- Track_data record ($48 bytes, 19 entries) ---
  Entries 0–15 : championship tracks in race order
  Entries 16–18: arcade Monaco variants (prelim, main, wet)
  Track_index is not bounds-checked; out-of-range reads garbage.

--- Track length (+$22 .w) ---
  Min: 329  (renderer computes length-169 and length-329; underflow is undefined)
  Max safe: 8188  (decompressed arrays are 2048 bytes; indexed by distance>>2)
  Hard absolute max: 65415  (sign arithmetic wraps at 65536; must stay positive)
  All 19 ROM tracks are in the range 3392–7744 (recommend staying ≤ 8000).
  Minimap position map must contain exactly (track_length >> 6) x/y byte pairs.

  Track lengths by name (all ROM tracks):
    Australia  6080   Monaco (champ)  6144   France        6144
    Brazil     6976   Canada          6720   Hungary       6464
    Portugal   6592   Spain           6784   Mexico        6848
    Gt Britain 6912   USA             7168   San Marino    7040
    Japan      7552   W. Germany      7488   Italy         7616
    Belgium    7744   Monaco prelim   3392   Monaco main   7616   Monaco wet  7616

--- Decompressed curve byte values ---
  $00        = straight
  $01–$2F    = left turn; $01 = extreme, $2F = softest (47 valid left values)
  $41–$6F    = right turn; $41 = softest, $6F = extreme (47 valid right values)
  $30–$3F    = INVALID — lower 6 bits ≥ 48 index past Road_displacement_table
  $70–$7F    = INVALID — same reason; game freezes / reads garbage pointer
  $80–$FF    = lap-wrap sentinel; game resets read pointer to track start (BPL branch)
  Road_displacement_table has one guard entry (index 0 = $FFA00000) and 47 real
  sub-arrays (indices 1–47). The sharpness index = byte & $3F; must be 0 or 1–47.

--- Curve RLE ROM encoding (in road_and_track_data.asm) ---
  Straight segment (3 bytes): $00, <length_lo>, $00
  Curve segment   (5 bytes): <length_hi>, <length_lo>, <curve_byte>, <bg_disp_hi>, <bg_disp_lo>
    length_hi can be non-zero (e.g. $02 = 512 steps added to lo-byte for 16-bit length)
  Terminator: $FF, $00
  bg_disp is a 16-bit signed background horizontal scroll delta applied per step.

--- Decompressed visual slope byte values ---
  Same encoding as curve bytes:
  $00        = flat
  $01–$2F    = downslope (extreme → soft)
  $41–$6F    = upslope (soft → extreme)
  $30–$3F, $70–$7F = INVALID (same table-index overflow risk as curves)
  $80+       = sentinel / lap-wrap

--- Decompressed physical slope byte values ---
  Signed byte, range –128 to +127; stored verbatim.
  Negative = downhill (reduces RPM drag via EXT.w / Update_rpm).
  Positive = uphill (increases RPM drag).
  ROM tracks use only $FF (downhill), $00 (flat), $01 (uphill).

--- Visual slope RLE ROM encoding ---
  Header: 1 byte of initial vertical background displacement (before first record).
  Flat segment    (3 bytes): <len_hi>, <len_lo>, $00
  Slope segment   (4 bytes): <len_hi>, <len_lo>, <slope_byte>, <bg_vert_disp_byte>
  Terminator: $FF

--- Physical slope RLE ROM encoding ---
  No header byte.
  Segment (3 bytes): <len_hi>, <len_lo>, <phys_slope_byte>
  Terminator: $FF (quirk: the terminator byte acts as the high byte of the final
  length, so 2 more bytes are consumed after it — last real entry must account
  for this; match the pattern used in existing tracks exactly).

--- Decompressed buffer sizes (RAM) ---
  Curve_data                         $00FF5B00  2048 bytes  (max 2047 steps + $FF)
  Background_horizontal_displacement $00FF6300  2048 bytes  (1024 word entries)
  Visual_slope_data                  $00FF7300  2048 bytes
  Physical_slope_data                $00FF8300  ≥2048 bytes (same index range)

--- Road renderer budget ---
  40 entries read ahead per frame for curve/slope look-ahead.
  96 interpolated display entries computed for road scanlines.
  224 scanlines in the tile-blit loop.
  Background displacement modulo: ANDI.w #$03FF (wraps at 1024 entries).

--- Sign data record format (RAM stream) ---
  4 bytes per record: distance.w, count.b, sign_id.b
  Terminated by $FFFF distance word (bit 15 set = end sentinel).
  Spawn trigger: sign spawns when (sign_distance − player_distance) < $0078 (120 units).
  Multiple signs in a group are spaced 16 distance units apart.
  Valid sign_id: 0–$14 for road signs (20-entry dispatch table in ai.asm).
  Finish line blocks all sign spawns when (track_length − player_distance) < 120.

--- Sign tileset stream format ---
  4 bytes per entry: distance.w, tileset_byte_offset.w
  Terminated by $FFFF.

--- Steering divisors (+$44 .l) ---
  Packed long: high word = Steering_divisor_straight, low word = Steering_divisor_curve.
  Must be non-zero (DIVS crash if zero).
  All championship tracks: $002B002B (43/43).
  Wet Monaco: $002F0038 (47/56).
  Adjusted by Steering_curve_divisors additive table (5 signed entries, default index 2 = 0,0).

--- Minimap position map ---
  Flat array of (x, y) signed byte pairs; no explicit length field.
  Must contain exactly (track_length >> 6) pairs.
  Example: San Marino length 7040 → need 110 pairs (220 bytes).

--- Lap targets format (15 × 3-byte BCD per track) ---
  14 real entries of <min_BCD>, <sec_BCD>, <centisec_BCD>.
  Entry 15: $99, $00, $00 sentinel.
  1 trailing $00 pad byte after sentinel.
  Total: 46 bytes per track.

--- General constraints summary ---
  1. track_length must be in range 329–8188.
  2. All decompressed curve/slope bytes must be in {$00} ∪ [$01–$2F] ∪ [$41–$6F]; never $30–$7F.
  3. Lap-wrap sentinels ($80+) must appear exactly once at the logical end of each stream.
  4. Both RLE streams (curve and slope) must fit in their 2048-byte decompressed buffers.
  5. Steering divisors must be non-zero.
  6. Sign records must be terminated by $FFFF and use only valid sign IDs (0–$14).
  7. Minimap must have exactly (track_length >> 6) entries.

--- Japan crossing research note (TRR-002, phase-1-safe) ---
  Repo evidence points to Japan as the best reference track for a grade-separated
  crossing, but the implementation split is only partly documented.

  Data-driven facts confirmed in the repo:
    - Track_data gives Japan only the standard per-track pointers: sign_data,
      sign_tileset, minimap_pos, curve_data, slope_data, and phys_slope_data.
      No dedicated bridge/crossing pointer field is present in track_config_data.asm.
    - minimap_pos is only signed (x, y) byte pairs. Compute_minimap_index in
      race_support.asm derives a byte index from distance>>5 with bit 0 cleared,
      then Load_minimap_position reads two bytes. There is no explicit z/layer field.
    - The extracted Japan track uses ordinary track streams plus its own sign_data
      and sign_tileset records, so at least some crossing presentation is selected
      from per-track data rather than a separate hard-coded track switch.

  Runtime-driven facts confirmed in the repo:
    - ai.asm routes special sign IDs through Parse_sign_data_Special and installs
      handlers from Special_sign_dispatch_table, so tunnel/approach behaviour is
      at least partly runtime-object driven.
    - rendering.asm switches the VDP to an H32 tunnel variant when
      Road_column_update_state == 2, proving that tunnel-style road presentation
      is not expressed by minimap data alone.
    - Aux_object_pool reserves slots for tunnel-style aux objects, so the runtime
      already has nontrivial support for tunnel presentation.

  Current phase-1 conclusion:
    - Japan proves that the game can present crossing-adjacent/tunnel-like track
      behaviour with the existing track data plus runtime handlers.
    - The repo does not yet document a generic overpass metadata model, generic
      bridge z-order flag, or minimap-layer field.
    - Map-first tooling work should therefore assume: zero-crossing tracks are the
      default, one grade-separated crossing is optional, and any later ASM work
      must be justified by a specific gap that tooling-only prototypes cannot cover.
  8. Lap targets must end with $99,$00,$00 sentinel + one $00 pad byte.

--- Championship Track Order ---
  Track_data table in src/track_config_data.asm has 19 entries × $48 bytes.
  Championship indices 0–15 (the 16-race World Championship season):
    0  San Marino      6  Canada         11  Mexico
    1  Brazil          7  Great Britain  12  Japan
    2  France          8  Italy          13  Belgium
    3  Hungary         9  Portugal       14  Australia
    4  West Germany   10  Spain          15  Monaco  ← championship final
    5  USA
  Arcade-only indices (Track_index_arcade_mode, offset +$0480 from Track_data base):
    16  Monaco Preliminary  (len 3392 — short qualifying race)
    17  Monaco Arcade Main  (len 7616)
    18  Monaco Arcade Wet   (len 7616, wet tyre steering $002F/$0038)

  Track_index ($FFFF9144 .w) counts 0–15 for championship races.
  Track_index == 16 ($0010) signals the championship is complete.
  Track_index_b ($FFFF9145 .b) is the low byte alias; used only by the password
  save serialiser (Save_player_state_to_buffer / Load_player_state_from_buffer).

  Use_world_championship_tracks ($FFFF9140): 0 = arcade mode, 1 = championship mode.
  When 0, Load_track_data uses Track_index_arcade_mode instead of Track_index,
  and applies the +$0480 offset to reach the three Monaco arcade entries.

      Initialize_results_screen, Assign_initial_rival_team,

--- Points System ---
  PointsAwardedPerPlacement (src/gameplay.asm):
    Placement:  1st  2nd  3rd  4th  5th  6th  7th+
    Points:      9    6    4    3    2    1    0

  Points are distributed by Accumulate_race_points, which reads placement
  order from Standings_team_order ($FFFF906E) and writes to Driver_points_by_team
  ($FFFF9030, 16 bytes — one byte per team).

  Sort_championship_standings: bubble-sorts 16 drivers by Driver_points_by_team.
  Sets Player_eliminated = $FFFF if player's score byte has MSB set (negative).

  Post_race_driver_target_points (src/gameplay.asm):
    16 teams × 2 bytes: [promote_threshold, partner_threshold]
    Example: top teams (Madonna/Rigel) require placement ≤ 2 to get a good rival;
    lower teams (e.g. Minarae, team 11) only need ≤ 8.
    Used by Championship_next_race_init to select music (Song 2 vs Song 3)
    based on whether player is on target.

--- Standings Order Buffer ---
  Initialize_standings_order_buffer builds Standings_team_order ($FFFF906E):
    1. PRNG-score each of 16 AI teams using Ai_performance_factor_by_team
       and Ai_performance_table (base scores + noise).
    2. Bubble-sort by score descending.
    3. Bias player slot to position 0 and rival slot to position 0 (lowest).
    4. Rotate buffer to actual finishing positions.
    5. Call Accumulate_race_points to distribute PointsAwardedPerPlacement.

  Ai_performance_factor_by_team (src/gameplay.asm): 16 bytes, multipliers 1–15.
    Team 12 (Rigel) has anomalously high factor — wild-card behaviour.
  Ai_performance_table (src/gameplay.asm): 16×8 bytes of base scores.

--- Elimination Logic ---
  Player_eliminated ($FFFFFCAC): 0=racing, 1=eliminated, $FFFF=not entered.
  Build_minimap_standings_row_buffer sets Player_eliminated = $FFFF if the
  player's team byte in Score_scratch_buf has MSB set (negative score).
  Sort_championship_standings also sets Player_eliminated on negative score.
  No explicit "must finish top N" rule is hard-coded per race — elimination is
  purely points-based (going negative in accumulated score).

--- Rival System ---
  Rival_team ($FFFF9042 .b), Player_team ($FFFF9043 .b) bit encoding:
    Player_team bits:
      [3:0] team index (0–15)
      [4]   "in team" / previous promotion flag
      [5]   rival race active
      [6]   year-2 championship leader flag (also livery swap trigger)
      [7]   player is champion (set by Championship_podium_check_champion)
    Rival_team bits:
      [3:0] rival team index
      [4]   rival pre-placed flag
      [5]   same-team variant flag
      [6]   rival promoted once
      [7]   rival promoted twice / new partner state

  Rival race triggers: Championship_next_race_init checks BTST #5, Player_team.
    Bit 5 set → rival race mode, uses rival-specific music + grid logic.
  Has_rival_flag ($FFFF914C?): set to 1 when rival race is active.

  Rival assignment (Assign_initial_rival_team, src/gameplay.asm):
    3 paths based on Player_team:
      1. Pre-assigned (team 8 Bullets): rival is next team above 8.
      2. No team yet: finds highest-scoring AI team from Driver_points_by_team.
      3. Has team: scans up to 4 slots above current rival team, skips slots
         already in Promoted_teams_bitfield ($FFFF9040 .w, 16-bit bitmask).

  Advance_rival_promotion_state: advances rival one tier per race if player
  is on target; uses stack-pop trampoline if all slots above are taken.

  Rival_grid_position computed by Init_race_player_state using:
    Rival_grid_base_table (16 bytes, base grid pos per team) +
    Rival_grid_delta_table (11 bytes: $FD $FE $00 $FF $FE $01 $01 $02 $01 $01 $00)

--- Driver/Team Maps ---
  InitialDriversAndTeamMap (src/gameplay.asm): 18 bytes, year-1 assignments.
    Player starts at Minarae (team 11); slot 11 holds Madonna as natural first rival.
  SecondYearDriversAndTeamsMap (src/gameplay.asm): 18 bytes, year-2 assignments.
    Player gets Madonna (team 0) with bit 6 set (year-2 flag); full roster reshuffle.
  Drivers_and_teams_map ($FFFF9044, 17 bytes): active driver→team mapping in RAM.

--- Password / Save System ---
  The "password" is a 64-character on-screen code displayed at Save_name_entry_init
  ($F288). The player writes it down and re-enters it via the password entry screen
  ($ED90). There is no hardware save; all state is encoded into these 64 characters.

  State serialised by Save_player_state_to_buffer (gameplay.asm ~$4283):
    Buffer: Player_state_save_buf ($FFFFFF80, 64 bytes)
    Bytes 0–36  (37 bytes): low 5 bits of Player_state_flags + Driver_points_by_team
                            [0..15] + Promoted_teams_bitfield + Rival_team +
                            Player_team + Drivers_and_teams_map[0..16]
    Bytes 37–55 (19 bytes): high bits (bits 5-7) of each byte pair from above,
                            packed as ((A>>5)&7) | ((B>>2)&$38)
    Byte 56: Track_index_b (which race is next, 0–15)
    Byte 57: mode discriminator:
               1 = resume at Championship_standings_init ($D6E2)
               2 = resume at Title_menu
               3 = resume at Championship_mode_init ($509C)
               4/else = null

  Checksum: Compute_save_data_checksum iterates buf[0..57] in 29 two-byte pairs,
    running EOR/rotate (ROXR) accumulation with $88/$10 XOR on carry.
    Result → 4 nibbles stored at Save_checksum_nibble_0..3 ($FFFFFFBC–$FFFFFFBF).
    Mismatch on restore → Demo_screen_transition (force title screen).

--- Notes for Championship Randomizer (RAND-009) ---
  - The 16-race order is fixed by Track_data table indices 0–15; changing race order
    requires reordering entries in Track_data (track_config_data.asm) or patching
    Load_track_data to use an indirection table.
  - Points system (9/6/4/3/2/1) is in PointsAwardedPerPlacement; safe to modify.
  - Post_race_driver_target_points controls rival-assignment and music; must remain
    consistent with actual game balance if teams are reshuffled.
  - Promoted_teams_bitfield uses bit positions matching team indices; if teams are
    renumbered by a randomizer, this bitmask logic must be updated accordingly.
  - The password save system encodes team indices directly; randomized team orders
    will break existing passwords (acceptable — document this limitation).
  - Player_eliminated is score-based only; no per-race minimum placement requirement
    means a player can theoretically survive by accumulating 1-pt finishes.
